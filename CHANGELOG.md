# Changelog

A concise list of every published version. For full release notes, see the corresponding commit on the v* tag.

## v0.45.2 вЂ” feat(0.45.2): dedup-before-create guard for stories that file GitHub issues вЂ” v0.45.1 stopped spec stories from filing issues for sibling failures, but left a residual case: when two or more stories legitimately have issue-filing in their acceptance (or a triage story re-runs on resume/retry), each could still create a fresh issue for the same root cause; `buildDefaultStoryPrompt` now instructs any agent whose acceptance requires opening an issue to FIRST run `gh issue list --state open --search "<symptom / file:line>"`, read the returned titles, and create only when no open issue matches (otherwise comment on the existing one) вЂ” and to give every created issue a specific, deterministic title (file + symptom) so later runs and sibling agents can match and skip; this makes the triage path idempotent across resume/retry. Note: the search-then-create check is not atomic across two agents filing in the same instant, but issue-filing is concentrated in the single closing triage story (depends on all others, runs alone at the last DAG level), so the realistic dup vectors вЂ” sibling spec stories (closed in 0.45.1) and triage re-runs (closed here) вЂ” are both covered; prompt-only change, no runtime behaviour change
## v0.45.1 вЂ” fix(0.45.1): parallel story agents file duplicate GitHub issues on a shared test failure вЂ” `buildDefaultStoryPrompt` (the default story prompt every agent receives) told agents to stay in scope and "mention unrelated bugs in the commit message", but said nothing about external side-effects; when a high-`--parallel` run had several stories whose acceptance includes running the full test suite (e.g. `npm test`), each agent independently hit the *same* failing spec produced by a *sibling* story and each ran `gh issue create`, so one real bug became N identical issues (observed: a 4-way dup of the same `GetShopsQueryFilter` numeric-city transform bug, filed by the DTO, mapper, service and triage stories within 4 minutes); the SCOPE DISCIPLINE block now forbids external side-effecting actions (opening issues, posting comments, pushing tags, sending notifications) unless the story's own acceptance criteria explicitly require it, and explicitly states that a `npm test` / `cargo test` / build FAILURE in a file the story did not create is neither its to fix nor its to report вЂ” it goes under `Noted (out of scope):` and a dedicated triage story (or the user) owns the dedup decision; prompt-only change, no Rust/orchestrator-runtime behaviour change, version bumped so npm re-syncs the bundled prompt
## v0.45.0 вЂ” release(0.45.0): refactor BaroEventForwarder god-class into per-event participants вЂ” the 400-line BaroEventForwarder class that inlined every BaroEvent translation (agent stream в†’ story_log, AgentState в†’ story_start/complete/error, AgentResult+CodexTurnEvent в†’ token_usage, ConductorState в†’ progress, Coordination+Critique в†’ story_log, FinalizeStarted+PrCreated в†’ finalize) is gone; replaced with six independently constructible `BaseObserver` subclasses under `packages/baro-orchestrator/src/participants/forwarders/`: `AgentStreamForwarder`, `StoryLifecycleForwarder`, `TokenUsageForwarder`, `ProgressForwarder`, `CoordinationForwarder`, `FinalizationForwarder` вЂ” each with a single bus subscription set and a single emit shape; barrel `forwarders/index.ts` exposes `joinBaroEventForwarders(env)` that wires all six, and callers wanting custom subsets can `new XxxForwarder().join(env)` directly; `orchestrate.ts` drops 280 LoC and goes from `new BaroEventForwarder().join(env)` to `joinBaroEventForwarders(env)`; OBSERVERS.md gets a new section per forwarder so the architecture doc tracks the code; this is the refactor [benchmarked in the blog post](https://jigjoy.ai/blog/claude-code-vs-codex-baro) вЂ” shipped via the hybrid baro run (Claude on Architect/Planner/Surgeon, Codex on the 9 Story Agents, Codex on Critic) at 3% Claude + 3% Codex of each provider's 5h subscription cap; PR #30; also drops 12 dead phase/demo/smoke scripts left behind from earlier development phases (`demo-single-story.ts`, `demo-multi-story.ts`, `phase{2,3,4}-demo.ts`, `e2e-test.ts`, `event-driven-demo.ts`, `smoke-mozaik-upgrade.ts`, `smoke-openai-{runner,critic-surgeon,story-agent}.ts`, `PHASE3-FINDINGS.md`) + their `demo:*` / `phase{2,3,4}` / `e2e` npm script entries; production-runtime scripts (`cli.ts`, `run-architect.ts`, `run-planner.ts`) kept untouched

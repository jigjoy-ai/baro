# Changelog

A concise list of every published version. For full release notes, see the corresponding commit on the v* tag.

## v0.57.1 — fix(0.57.1): worktree dependency sharing — installs are visible across stories on any stack. Worktree isolation previously only symlinked dep dirs (`node_modules`, `.venv`, `vendor`) that ALREADY existed at the repo root, so on a fresh clone (no `node_modules` yet) the setup story installed into its own worktree and dependent stories couldn't see it — their tests failed with bare "command failed". Now dep locations are discovered manifest-driven (`package.json`/`pyproject.toml`/`composer.json`/…) at any depth — including monorepo subdirs like `frontend/` — the shared dir is pre-created so the first install writes through the symlink into one location every worktree sees, and commit-exclusion matches dep dirs at any depth. Replicates the old single-shared-workspace behavior. `runner.mjs` rebuilt.

## v0.57.0 — feat(0.57.0): `baro login` + tokenless `baro connect`. Sign in (or sign up) straight from the terminal: `baro login` opens the browser once, you confirm a short code, and a long-lived credential is stored under `~/.baro/credentials.json`. After that `baro connect` needs **no `--token`** — it self-registers a runner from that credential, so you never visit the dashboard or paste a pairing token. First sign-in grants the same starter credits as the web flow. (Browser-based device flow against the control plane's new `/cli/auth/*` + `/cli/runners/register` endpoints.) `runner.mjs` rebuilt.

## v0.55.4 — feat(0.55.4): `baro connect` — stable per-machine runner identity. The runner now registers with a stable `runnerId` (the machine `hostname`, overridable via `RUNNER_ID`) and sends its `hostname`, instead of a per-process `runner-<pid>` that changed on every reconnect. This lets baro-cloud show one runner across reconnects + keep history (online / last-seen) rather than a new entry each time. `runner.mjs` rebuilt.

## v0.55.3 — feat(0.55.3): `baro connect` — run on a chosen GitHub repo + open a PR. When the control plane dispatches a run with a `repo` + `githubToken` (the user picked a repo in the baro-cloud dashboard), the runner clones that repo into a temp dir with token auth, runs `baro --headless` there, and exports `GH_TOKEN`/`GITHUB_TOKEN` so baro pushes the branch + `gh pr create` opens the PR — then removes the clone. Without a repo it still runs in the local `--workspace` dir. `runner.mjs` rebuilt.

## v0.55.2 — fix(0.55.2): `baro connect` — keep opus for planning + stream the planning phase. Reverts 0.55.1's sonnet planning default: the Architect + Planner stay on baro's own routing (the latest `opus` for the `claude` backend) — quality planning is the point, and an opus outage is transient + now visible (see 0.55.1's stderr-tail error reporting) rather than something to paper over with a weaker model. Headless mode also now prints `architect_start` / `architect_complete` / `plan_ready` events to stdout, so a remote runner / the baro-cloud dashboard shows the **planning** phase live, not just execution. `runner.mjs` rebuilt.

## v0.55.1 — fix(0.55.1): `baro connect` — surface the failure cause + robust planning. The runner used to discard baro's stderr, so a failed run showed up in baro-cloud as a bare exit code with no reason; it now keeps the tail of stderr and returns it as the run's `error` (e.g. "planner FAILED: claude … --model opus"). It also defaults the Architect + Planner to a **less rate-limited model** (`sonnet`, overridable via `BARO_PLAN_MODEL`) — those phases were always on `opus`, so a run died whenever opus was throttled or temporarily unavailable; planning now survives that. Story tiers are unchanged. `runner.mjs` rebuilt.

## v0.55.0 — feat(0.55.0): `baro connect` — run this machine as a baro-cloud runner. `baro connect --token <rt_…> [--workspace <git repo>]` pairs the machine with [baro-cloud](https://app.baro.jigjoy.ai) and runs each goal dispatched to it via `baro --headless` **over your Claude subscription** (no API billing — a stray `ANTHROPIC_API_KEY` is stripped from the child), streaming events back. It's a thin daemon: a new bundled `runner.mjs` (vendored wire protocol + `ws`) connects to the control plane, registers with the pairing token, and on each dispatch spawns `baro --headless "<goal>"` and forwards its native event stream; it auto-reconnects and answers keep-alive pings. The Rust side handles `connect` **before** clap / the session lock — a small `run_connect` that locates `runner.mjs` (same discovery as `cli.mjs`; staged to `~/.baro/bin/` by postinstall), spawns it with `RUNNER_TOKEN` / `WORKSPACE_DIR` / `CONTROL_URL` / `BARO_BIN` (this binary), and waits. The TUI and every other command are untouched. Verified end-to-end against the live control plane (connect → register → dispatched goal → PR). `cargo build -p baro-tui` clean.

## v0.54.0 — feat(0.54.0): `--headless` mode — run baro from a goal with no TUI, for CI / automation / remote runners. `baro --headless "<goal>"` runs the full pipeline (Architect → Planner → execute) without the terminal UI: it **auto-confirms the plan** (no interactive Review keypress) and **streams the orchestrator's native event JSON to stdout**, exiting when the run finishes (non-zero on a planning/branch failure). The TUI is just one consumer of the run's event channel, so the implementation reuses **all** config + planning unchanged — `run_app` takes an `Option<&mut Terminal>` and only the event loop branches (headless skips the terminal + key thread, auto-confirms on `PlanReady` via `confirm_and_execute`, exits on `OrchestratorExited`); `orchestrator_client` gains an `echo_raw` flag that tees each native event line to stdout. Requires a goal argument; `--headless --llm openai` requires `OPENAI_API_KEY`. This is the seam a remote runner (baro-cloud) uses to run goal→PR autonomously over the user's subscription. TUI path unchanged; `cargo build -p baro-tui` clean. (PR #58.)

## v0.53.0 — feat(0.53.0): pluggable story execution via a `StoryExecutor` seam. Story execution is now injectable. `StoryFactory` resolves each story's route + worktree cwd and **delegates** to a `StoryExecutor` interface instead of constructing the backend agent inline; the historical in-process behaviour moves **verbatim** into `LocalStoryExecutor` (the default), so the claude/codex/openai/opencode/pi paths are byte-for-byte unchanged. Pass `OrchestrateConfig.executor` (or `StoryFactoryOptions.executor`) to run a story's agent loop somewhere other than in-process — a mock for tests, or an out-of-process / remote executor. The Conductor / Planner / Critic / Surgeon / Sentry / Librarian / Cartographer can't tell the difference: every executor must deliver the agent's bus events, including the terminal `StoryResult`, onto the bus. The seam is now part of the public API — `StoryExecutor`, `StoryExecution`, `StoryExecOpts`, `LocalStoryExecutor` (plus `StoryRoute` / `Backend` / `StorySpawnRequestData`) are exported from the package entry so embedders can implement and inject one. Pure refactor — no behaviour change to the local path. New `test/story-executor.test.ts` proves an injected executor is delegated to + disposed and that the default is `LocalStoryExecutor`. Orchestrator suite **54/54**, `tsc --noEmit` clean, `cargo build -p baro-tui` clean. (PR #57.)

## v0.51.4 — fix(0.51.4): real per-story git worktree isolation — parallel stories no longer share one working tree — closes the HIGH-severity bug (#50) where every parallel story agent ran in the **same** `cwd`, so they shared one working tree and one git index: `git add -A` in one story swept up siblings' in-flight edits, concurrent commits raced on `.git/index.lock`, and the codex `--dangerously-bypass-approvals-and-sandbox` flag was justified on "per-story git worktrees" that **were never created** (`git worktree` appeared in 0 lines of source). New `WorktreeManager` (`packages/baro-orchestrator/src/worktree.ts`): each story runs in its own `git worktree` on branch `baro-wt/<runId>/<storyId>` created off the run-branch HEAD, edits + commits in that isolated tree, and on success merges back onto the run branch in the main worktree under the existing `GitGate` mutex — so `git add -A` is safe again without rewriting any agent prompt, and the codex sandbox-bypass rationale becomes true. Because `@mozaik-ai/core` delivers events to handlers **without awaiting them**, a promise-chain mutex was added around `Conductor.handle()` so two `StoryResult`s settling together can't interleave and spawn the next DAG level before a merge-back finishes — guaranteeing each level-N merge-back lands before level-N+1 stories branch off HEAD (dependencies resolve correctly). Conflicts between same-level stories auto-resolve with `git merge -X theirs` (the merging story wins) + a loud warning — strictly better than the old silent last-writer-wins; an unresolvable merge **preserves** the story branch for recovery instead of discarding the work. Dependency dirs (`node_modules`/`.venv`/`vendor`) are best-effort symlinked into each worktree so builds/tests resolve; the auto-commit safety net (for agents that pass without committing) never commits those symlinks. The run branch is pushed **once at the end** (worktree merge-backs only touch the local branch) rather than per story, so a slow network push can't stall the Conductor via the shared git gate. New `onStoryFailed` hook discards a failed story's worktree; `cleanupAll` sweeps stragglers + the temp dir on exit. Escape hatch: `withWorktrees` config flag (default on with git) or the `BARO_NO_WORKTREES` env var (NO_COLOR-style — set to any value to disable). Hardened over a six-round Greptile review (spawning-set try/finally, robust git exit-code handling, preserved-branch cleanup, merge-abort/staging failures surfaced) ending at "safe to merge". Ships a deterministic test suite (`test/worktree.test.ts`, **real git in a temp dir, no LLMs**) covering lifecycle, isolation, conflict resolution, the safety net, dep-dir symlinks, fallback, and cleanup; verified end-to-end with a live `--llm claude` run (two parallel stories in isolated worktrees, both merged back cleanly). Orchestrator suite **42/42**, `tsc --noEmit` clean. **Closes #50, PR #53.**
## v0.51.3 — fix(0.51.3): semantic memory recall now works on subprocess backends — long-lived reader reloads the on-disk index — by @tcsenpai (TheCookingSenpai) — closes the HIGH-severity bug where cross-agent semantic memory (the 0.49.0 headline feature) silently recalled **0** findings on the subprocess backends (codex / opencode / pi): findings WERE written to disk, but the reader never saw them, so those agents paid the memory overhead for zero benefit. Root cause: `VectraMemoryStore` (`packages/baro-memory/src/vectra-store.ts`) holds a Vectra `LocalIndex` created **once** at orchestrator startup; Vectra loads the index into a private in-memory `_data` on first query (when empty) and **never reloads**, while story agents in **separate processes** write findings to the same on-disk `index/index.json` via the `baro-memory` CLI — so the long-lived reader's `recall()` / `gatherContext()` / `getStats()` queried a frozen empty snapshot → `totalFindings = 0`, `recall → []` forever. (0.50.1 fixed *loading*; this is why it still did nothing once loaded.) Fix: a new `refreshIndexIfChanged()` stats `index.json`'s `mtimeMs` and, when it advances, swaps `this.index` for a fresh `new LocalIndex(...)` (Vectra exposes no public reload; a new instance lazily loads current on-disk data on its next query) — throttled by mtime so it only re-reads when the index actually changed (cheap vs an LLM turn), wrapped in try/catch to keep the current index on any error, and called **only on the read paths** (`recall` + `getStats`; `gatherContext` delegates to `recall`) since writers already refresh Vectra's `_data` via `beginUpdate` and reloading mid-write could clobber a pending batch. Greptile P2 hardening folded in before merge: the new `LocalIndex` is constructed **before** the `lastIndexMtimeMs` watermark is advanced, so a failed instantiation retries on the next call instead of locking the reader onto a stale snapshot for that mtime. Ships a regression test reproducing the exact broken ordering (a reader opened **before** an external write must recall the finding after it) — verified red without the fix, green with it. `tsc --noEmit` clean · `vitest run` **26/26**. **Contributed by @tcsenpai — PR #52, closes #51.** Thank you!
## v0.51.2 — refactor(0.51.2): CLI parsing + session-lock extracted into a dedicated `cli` module — by @kisshan13 — `main.rs` had grown to own both argument parsing and the `baro.lock` session-lock lifecycle inline; this lifts them out into `crates/baro-tui/src/cli/` (`cli.rs` for the clap `Cli` struct + `parse()`, `session.rs` for `SessionLock`, `mod.rs`), leaving `main.rs` to just call `cli::cli::parse()`. ## v0.48.0 — feat(0.48.0): per-story model tiering across all backends, with per-story OpenAI endpoints — model selection used to be per *phase* (`--llm` / `--story-llm`): every story in a run shared one backend and one model. baro now tiers per *story*. The Planner classifies each story by blast radius — "what breaks if an agent gets this wrong?" — and tags it `haiku` (mechanical / self-contained), `sonnet` (one contained module), or `opus` (cross-cutting, schema, wiring, a DAG hub). A new `--tier-map` binds those semantic tiers to concrete `backend:model` routes (e.g. `haiku=openai:MiniMax-M3,sonnet=openai:MiniMax-M3,opus=claude:opus`), so a single DAG can route cheap single-concern stories to one backend and high-stakes stories to another. A story's route can name any of the three backends (`claude:…`, `openai:…`, `codex:…`); the StoryFactory now picks the agent class per story rather than from the global `--llm`. An `openai:` route can additionally name its **own endpoint** with `@` — a registered name via `--openai-endpoint name=url` (repeatable) or an inline `@https://…` URL — so one run can hit several OpenAI-compatible endpoints at once (e.g. MiniMax + real OpenAI); the runtime keeps one cached client per endpoint instead of a single env-bound global. API keys never travel on the command line: each endpoint resolves its key from `BARO_OPENAI_KEY_<NAME>`, falling back to `OPENAI_API_KEY`.
